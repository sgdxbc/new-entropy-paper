\paragraph{Preliminary on verifiable random function (VRF)~\cite{vrf,vrf-rfc}.}
A VRF provides two algorithms, $\VRFGen$ and $\VRFVerify$.
$\VRFGen$ takes a secret key $\sk$ and an input $x$, and outputs a pseudorandom value $y$ and a proof $\pi$.
$\VRFVerify$ takes a public key $\pk$, an input $x$, a value $y$, and a proof $\pi$, and outputs a boolean indicating whether $y$ is the correct VRF output for $x$ under $\pk$.
In our setting, $x$ is the 32-byte hash of a data block.

A VRF satisfies the following properties:
\begin{itemize}
    \item \textbf{Pseudorandomness:} For any input $x$, the output $y$ is indistinguishable from random to any efficient adversary that does not know the secret key $\sk$.
    \item \textbf{Uniqueness:} For any input $x$, there is a unique output $y$ that can be generated by $\VRFGen$ with the corresponding proof $\pi$.
    \item \textbf{Verifiability:} Given the public key $\pk$, input $x$, output $y$, and proof $\pi$, anyone can efficiently verify that $y$ is the correct VRF output for $x$ under $\pk$ using $\VRFVerify$.
\end{itemize}

\subsection{Sampling to Defend Against Byzantine Nodes Globally}
\label{sec:approach:sample}

We observe a common pattern in DSNs with $O(P)$ fault tolerance.
In these systems, placement groups are formed independently, and fault tolerance is analyzed locally for each group, resembling centrally managed storage practice.
This approach is effective for simultaneous failures of rational nodes, but it becomes inadequate in the presence of Byzantine nodes.
Unlike benign failures, which are tied to specific placement groups, Byzantine adversarial power is global and can be distributed arbitrarily across groups.
In such DSNs, each placement group is secured against Byzantine behavior independently, which leads to redundant defense.
In fact, to tolerate attacks on individual groups, these DSNs have to enlarge placement groups so that they can even tolerate an attack that targets all groups simultaneously, which is clear over-provisioning with undesirable trade-offs.
To tolerate Byzantine nodes more efficiently, data placement must defend against Byzantine power globally rather than independently within each placement group.

Among prior DSNs, Swarm is the only system that tolerates $O(N)$ Byzantine nodes, enabled by its disjoint neighborhoods.
This disjointness is key to global defense: a Byzantine node can reside in only one neighborhood at a time.
Therefore, with \nicefrac{1}{3} Byzantine nodes globally, at most \nicefrac{1}{3} of placement groups are compromised, regardless of the adversarial distribution.
Unlike DSNs with $O(P)$ fault tolerance, Swarm does not require each placement group to withstand the full adversarial budget, so the group size need not scale with the total number of nodes.

However, sacrificing placement groups demands manual redundancy repair across placement groups and additional redundancy, as discussed in \cref{sec:motivation:review}.
% As discussed in \cref{sec:motivation:review}, redundancy loss from failed placement groups can be repaired only manually, if it can be repaired at all.
% It also requires an additional erasure-coding layer, which introduces extra storage overhead on top of already costly in-neighborhood replication.
To avoid these pitfalls, each placement group in our system must remain uncompromised even under global Byzantine pressure.
Our data placement solution allows defensive effort to be distributed across groups as flexibly as Byzantine attacking power.
Each placement group receives a level of defense that dynamically matches the attack it faces.
When attacks concentrate on specific groups, the DSN correspondingly concentrates fault tolerance on those groups to ensure their safety.

To realize this goal, we shift from a data-centric approach to a node-centric one, illustrated in \cref{fig:approach}.
In prior DSNs, incentives are tied to data blocks and their placement groups: payment originates from the client storing the data and is distributed to the responsible nodes.
Although this model is straightforward, it prevents placement groups from sharing defensive power because each group is governed by a separate incentive fund.
In \sys, incentives paid for all stored data are pooled and distributed directly to nodes.
Nodes are rewarded uniformly according to the number of placement groups they participate in, regardless of which specific groups they join.
This node-centric allocation is exactly what we need.
The incentive captured by Byzantine nodes can be interpreted as a global defensive budget that effectively ``tracks'' Byzantine behavior across groups.
Because each node's incentive is independent of others, the reward of a rational node in a placement group is not influenced by Byzantine nodes joining the same group.
As a result, rational nodes remain in place, placement groups remain effective, and data availability is preserved.

Our node-centric approach introduces new challenges.
A key benefit of the data-centric approach is that incentives naturally balance rational nodes across placement groups, since each data block offers comparable rewards and attracts a comparable number of nodes.
Under a node-centric design, this self-balancing no longer occurs, so balancing becomes the responsibility of the data placement mechanism.
The mechanism must assign an appropriate number of rational nodes to each data block.
We use VRFs to randomly sample nodes for placement groups, and we refer to sampled nodes as endorsed nodes, yielding a uniform and publicly verifiable assignment.
Under this sampling process, each node is endorsed and rewarded independently, which enables node-centric incentives.
Because VRF outputs are verifiable on-chain, nodes can profit only from placement groups that endorse them, and rational nodes maximize rewards by participating in all such groups.
The expected number of rational nodes per placement group is uniform and can be tuned through the sampling rate according to node failure rates.

We now detail the sampling scheme in \sys, inspired by Algorand~\cite{algorand}.
Given a data block, each node runs $\VRFGen$ with its secret key and the data hash as input $x$, obtaining pseudorandom output $y$.
The node is endorsed for the placement group if $\nicefrac{y}{Y}<p$, where $p$ is a public target sampling rate; endorsed nodes are eligible to store redundancy and earn rewards in that placement group.
In other words, for a data block with hash $h$, the placement group $\mathcal{P}$ is
$$\mathcal{P}(h)=\left\{n\in\mathcal{N}\mid\frac{\VRFGen(\sk_n,h)}{Y}<p\right\}$$
where $\mathcal{N}$ is the set of all nodes and $Y$ is the maximum VRF output.
Beyond scalable fault tolerance, this approach is also algorithmic: the only consistent state it requires is the target sampling rate $p$, which can be inferred from blockchain history, analogous to Bitcoin difficulty.
Thus, sampling-based data placement can achieve both strong resiliency and scalable on-chain footprint.

\subsection{Sampling for Erasure Coding}
\label{sec:approach:ec}

We now turn to the final goal of \sys{}: scalable storage capacity that grows with total storage supply and remains independent of placement-group size.
To achieve this, we employ erasure coding (EC)~\cite{rs-code} within placement groups to bound redundancy regardless of group size.
In prior DSNs that use EC~\cite{filecoin,sia,shelby,walrus}, encoded-fragment assignments are recorded on-chain.
Data blocks are encoded into a fixed number of fragments, and both fragment commitments and node assignments are maintained as on-chain state.
Endorsed nodes then store the assigned fragments and prove storage against those commitments.
In \sys, we instead require an algorithmic approach, because storing fragment-to-node assignments on-chain leads to either high on-chain overhead~\cite{filecoin,sia,shelby} or fixed membership~\cite{walrus}.

Algorithmically assigning fragments in dynamic placement groups of \sys is challenging.
To ensure long-term recoverability, each endorsed node must be assigned distinct encoded fragments, so that any set of $k$ rational nodes can recover the data block, where $k$ is the repair threshold of the EC scheme.
Because node failures are random and occur in nondeterministic order, assignments for newly joined nodes must adapt to previous failures.
Off-chain coordination of this assignment is infeasible, because a Byzantine majority could manipulate assignments to induce data loss.

We therefore also use sampling to assign encoded fragments to endorsed nodes.
The hash of proof $\pi$ is interpreted as a seed for sampling the fragment to be stored.
Each node can independently compute its assigned fragment from the proof, and the assignment is pseudorandom and publicly verifiable.
However, this assignment is not guaranteed to be injective, so multiple nodes may be assigned the same fragment, and some fragments may not be assigned at all.
We solve this problem by sampling multiple fragments for each node to ensure that every fragment is assigned to at least one node with high probability.

We further improve the space efficiency of this sampling approach by leveraging the rateless property of verifiable rateless EC~\cite{verify-rateless-ec}.
A rateless EC can generate an unbounded number of encoded fragments from an original data block, where each fragment can be interpreted as the XOR of sampled source fragments under recoverability-preserving sampling distributions.
This property lets us assign one distinct fragment to each node with high probability by directly interpreting the hash of proof $\pi$ as the fragment index, thereby eliminating the need to sample multiple fragments per node and reducing storage overhead.

However, the rateless property also implies a potentially unbounded set of fragments stored by endorsed nodes.
Their storage cannot be proven through conventional on-chain fragment commitments, because that would require maintaining an unbounded number of commitments.
Verifiable rateless EC provides an $\ECVerify$ algorithm that validates a fragment using the original data-block hash and the fragment index~\cite{verify-rateless-ec}.
As a result, the smart contract needs to store only the original data-block hash on-chain while still being able to verify storage for all possible fragments.
To prove correct storage of an assigned fragment, a node submits the fragment together with its VRF output and proof.
The smart contract first runs $\VRFVerify$ to validate endorsement, then derives the assigned index from the proof and runs $\ECVerify$ to validate fragment integrity.
The node is rewarded only if both checks pass.
