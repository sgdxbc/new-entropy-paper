\subsection{Preliminary}
\label{sec:approach:preliminary}

\paragraph{Verifiable random function (VRF)~\cite{vrf,vrf-rfc}.}
A VRF provides two algorithms, $\VRFGen$ and $\VRFVerify$.
$\VRFGen$ takes a secret key $\sk$ and an input $x$, and outputs a pseudorandom value $y$ and a proof $\pi$.
$\VRFVerify$ takes a public key $\pk$, an input $x$, a value $y$, and a proof $\pi$, and outputs a boolean indicating whether $y$ is the correct VRF output for $x$ under $\pk$.
In our setting, $x$ is the 32-byte hash of a data block.

A VRF satisfies the following properties:
\begin{itemize}
    \item \textbf{Pseudorandomness:} For any input $x$, the output $y$ is indistinguishable from random to any efficient adversary that does not know the secret key $\sk$.
    \item \textbf{Uniqueness:} For any input $x$, there is a unique output $y$ that can be generated by $\VRFGen$ with the corresponding proof $\pi$.
    \item \textbf{Verifiability:} Given the public key $\pk$, input $x$, output $y$, and proof $\pi$, anyone can efficiently verify that $y$ is the correct VRF output for $x$ under $\pk$ using $\VRFVerify$.
\end{itemize}

We observe a common pattern of DSNs with $O(P)$ fault tolerance.
In these DSNs, the placement groups are formed independently, and the fault tolerance reasoning is local to each placement group, somewhat resemble to the centrally managed storage practice.
While this approach is effective on handling simultaneous failures of the rational nodes, it falls short after introducing Byzantine nodes.
Unlike the benign failures, which can be considered as bound to specific placement groups, the Byzantine nodes are global attacking power that may arbitrarily distribute across the placement groups.
In those DSNs, every placement group is secured against Byzantine nodes independently, and an $O(N)$ placement group size can actually defend Byzantine nodes to attack every placement group at the same time, which is an obvious overkill and leads to various undesirable trade-offs.
To tolerate Byzantine nodes more efficiently, the data placement solution must eliminate the redundant defending efforts by defending the Byzantine nodes globally rather than individually from placement groups.

Among previous DSNs, Swarm is the only one to tolerate $O(N)$ Byzantine nodes thanks to its disjoint neighborhoods.
The disjoint structure of placement groups is the key to defending against Byzantine nodes globally: a Byzantine node can only reside in one neighborhood at a time.
Therefore, assuming \nicefrac{1}{3} Byzantine nodes globally, at most \nicefrac{1}{3} of placement groups are compromised regardless of how Byzantine nodes are distributed.
Unlike in DSNs with $O(P)$ fault tolerance, in Swarm no placement group needs to defend the entire attacking power, so the placement group size does not need to scale with the total number of nodes.

However, sacrificing placement groups is never ideal.
As discussed in \cref{sec:motivation:review}, redundancy loss caused by failed placement groups can only be repaired manually, if possible at all.
It also requires an additional erasure-coding layer, which adds more storage overhead on top of the already costly replication within neighborhoods.

To avoid these pitfalls, we must have each of our placement group to be able to defend all Byzantine nodes and never compromise.
With our data placement solution, the shared defending effort of Byzantine nodes should be able to be flexibly distributed across placement groups just as the attacking power of Byzantine nodes.
The placement groups will receive dynamic defending effort matching the attacking power applied to them.
As the attacking concentrate on individual groups, the DSN also concentrate its fault tolerance on them to ensure they are not compromised.

To realize this goal, our key principle is to switch from a data-centric approach to a node-centric one.
In prior DSNs, incentive belongs to data blocks and their placement groups: it originates from the client storing the data and is received by the nodes participated into storing it.
This idea is straightforward but forbid placement groups to share defensive efforts because they are driven by different incentives.
In \sys, incentive paid for all stored data is pooled and directly distributed to nodes.
The nodes are uniformly rewarded based on the number of placement groups they participate, regardless of which specific placement groups are those.
The node-centric incentive allocation perfectly achieve our goal.
The (wasted) incentive received by the Byzantine nodes can be treated as the global defensive effort and will ``track down'' the Byzantine nodes throughout the system no matter which placement groups they stay in.
By ensuring every node having its own incentive isolated to the others, the incentive received by each node in the same placement group does not interfere with each other anymore, so as nodes' behavior: no matter how many additional Byzantine nodes clustering in a placement group, the rational nodes of the group receive unaffected incentive and will remain in the group.
Thus, the placement groups will stay effective, and the data availability is secured.

Our node-centric approach brings new challenges.
The benefit of data-centric approach is that rational nodes are automatically balanced across placement groups driven by incentive, as fair incentive of each data block will attract fair number of nodes.
With node-centric design, the rational nodes will not balance themselves, and the responsibility shifts to the data placement solution.
The data placement solution must assign certain number of rational nodes to each data block with incentive.
In another word, the nodes are only rewarded for storing the assigned data blocks.
We employ VRF to randomly sample nodes for the placement groups, which we call endorse the nodes, to enable a uniform and publicly verifiable assignment.
Under this random sampling, every node is independently endorsed and rewarded, thus enable node-centric incentive.
Thanks to VRF, the endorsement can be validated on-chain, so the nodes can only make profit from placement groups endorsing them, and the rational nodes will maximize their rewards by participating all of them.
The number of rational nodes in the placement groups will be uniform and can be tuned according to their failure rate by configuring the sampling rate.

Now we detail the sampling scheme in \sys, inspired by Algorand~\cite{algorand}.
Given a data block, every node runs $\VRFGen$ with its secret key and the data hash as the input $x$ and obtains the pseudorandom output $y$.
The node is endorsed for the placement group if $\nicefrac{y}{Y}<p$, where $p$ is a public target sampling rate; endorsed nodes are eligible to store redundancy and earn rewards in that placement group.
In other words, for a data block with hash $h$, the placement group $\mathcal{P}$ is
$$\mathcal{P}(h)=\left\{n\in\mathcal{N}\mid\frac{\VRFGen(\sk_n,h)}{Y}<p\right\}$$
where $\mathcal{N}$ is the set of all nodes and $Y$ is the maximum VRF output.
Besides scalable fault tolerance, this approach is also algorithmic: the only consistent state it requires is the target sampling rate $p$, which can be inferred from blockchain history, analogous to Bitcoin difficulty.
Thus, sampling-based data placement can achieve both strong resiliency and scalable on-chain footprint.

\subsection{Sampling for Erasure Coding}
\label{sec:approach:ec}

In \sys, each placement group contains a dynamic set of nodes that continuously join and leave.
To ensure recoverability of data over time, we need to assign every encoded fragment \lijl{We haven't discussed encoding yet.} to at least one node in the placement group, and the fragment assigned to new nodes must be dynamically determined according to the previous failures.
Storing the assignment of fragments to nodes on-chain would lead to large on-chain overhead.
It is infeasible for nodes to coordinate this assignment off-chain, because a Byzantine majority could manipulate the assignment to induce data loss.
\lijl{We should define the problem and the desired properties clearly. Data is encoded into some fragments (fixed number?). Each endorsed node stores some fragments. Need to ensure the number of unique fragments on non-faulty nodes to exceed a certain number, without storing extra metadata on-chain.}

We propose to also use the sampling approach to assign random encoded fragments to the endorsed nodes.
The hash of proof $\pi$ is interpreted as a seed for sampling the encoded fragment to store.
Every node can independently compute its assigned fragment from the proof, and the assignment is pseudorandom and publicly verifiable.
However, this assignment is not guaranteed to be injective, so multiple nodes may be assigned the same fragment, and some fragments may not be assigned at all.
We solve this problem by sampling multiple fragments for each node to ensure that every fragment is assigned to at least one node with high probability.

We further optimize the sampling approach for erasure coding by leveraging the rateless property of verifiable rateless EC~\cite{verify-rateless-ec}.
\lijl{Should first explain what we are optimizing by using rateless code.}
A rateless EC can generate an unbounded number of encoded fragments from the original data block, whose content can be interpreted as the XOR of sampled encoded fragments, using different sampling distributions to ensure recoverability.
This allows us to assign distinct fragments to nodes with high probability by directly interpreting the hash of the proof $\pi$ as fragment index.
Rateless EC also allows recovery of the original data block from the encoded fragments stored by $k$ nodes, and it improves space efficiency by eliminating the need to sample multiple fragments per node.

Last but not least, the verifiable rateless EC provides a $\ECVerify$ algorithm to validate the integrity of an encoded fragment from the hash of the original data block and the fragment index, which is essential for the smart contract to verify the correct storage of potentially infinite different fragments.
\lijl{Is this only a challenge for rateless EC? Or also for other coding scheme if being used in our context?}
Only the hash of the original data block needs to be stored on-chain.
To prove correct storage of an assigned fragment, a node submits the fragment together with its VRF output and proof.
The smart contract first runs $\VRFVerify$ to validate endorsement, then derives the assigned index from the proof and runs $\ECVerify$ to validate fragment integrity.
The node will be rewarded if both checks are passed.
