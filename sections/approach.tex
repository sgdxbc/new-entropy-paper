\paragraph{VRF definition and properties.}
A VRF provides two algorithms, $\VRFGen$ and $\VRFVerify$.
$\VRFGen$ takes a secret key $\sk$ and an input $x$, and outputs a pseudorandom value $y$ and a proof $\pi$.
$\VRFVerify$ takes a public key $\pk$, an input $x$, a value $y$, and a proof $\pi$, and outputs a boolean indicating whether $y$ is the correct VRF output for $x$ under $\pk$.
In our case, $x$ is 32-byte array data hash.

A VRF satisfies the following properties:
\begin{itemize}
    \item \textbf{Pseudorandomness:} For any input $x$, the output $y$ is indistinguishable from random to any efficient adversary that does not know the secret key $\sk$.
    \item \textbf{Uniqueness:} For any input $x$, there is a unique output $y$ that can be generated by $\VRFGen$ with the corresponding proof $\pi$.
    \item \textbf{Verifiability:} Given the public key $\pk$, input $x$, output $y$, and proof $\pi$, anyone can efficiently verify that $y$ is the correct VRF output for $x$ under $\pk$ using $\VRFVerify$.
\end{itemize}

\subsection{Desired Properties of Data Placement}

Data placement must only persist on-chain metadata whose size is independent to the placement group size.
Otherwise, in the case where large placement groups are employed to tolerate high failure rates, the on-chain metadata would be impractically bloated.
In some sense, the placement must be stateless, i.e., the placement of a data block can be determined solely based on the block's content and the current state of the system, without relying on any additional information or history.

Data placement must be compitable with erasure coding.

A placement group must include a sufficient number of non-Byzantine nodes.
It also must provide a mechanism to verify that the nodes in the placement group have correctly stored the data blocks.

Data placement must make repair efficient.
It must be easy to detect redundancy failure for a data block.
It must be possible and efficient to introduce new nodes to the placement group to replace failed nodes.
