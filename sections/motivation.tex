\section{Motivation}
\label{sec:motivation}

\subsection{DSNs and Their Data Placement}

Decentralized storage networks (DSNs) provide blob storage service backed by decentralized storage resources collectively provided by permissionless nodes.
Some nodes are rational and aim to generate profit with their storage supply, while the other nodes are Byzantine, behaving arbitrarily and potentially maliciously.
DSNs can provide reliable storage service, ensuring the stored data can be successfully retrieved, given that there are at most a certain fraction (e.g. \nicefrac{1}{3}) of nodes are Byzantine.
They guarantee reliable storage by driving rational nodes to correctly store and serve the data with incentive and tolerating Byzantine nodes.

In this work, we focus on improving the data placement strategy in DSNs.
In storage systems, data placement means given a data block, how to determine which nodes should store redundancy for this data block and what kind of redundancy they store.
These storing nodes are called the placement group of the data block.
Storing data redundantly on multiple nodes is crucial for tolerating simultaneous failures across them.
For example, by storing $r$ copies of data on distinct nodes, the data will still be available even when $r-1$ of the nodes fail at the same time.
So, as long as the storage system can repair the redundancy before all $r$ nodes fail, the storage will be durable.

In practice, storing large number of data copies incur excessive storage overhead.
Storage systems commonly employ erasure coding to achieve better storage efficiency.
They encode a data block into $n$ fragments, any $k$ of which can recover the data block, and store them on $n$ nodes.
Then, the storage systems can tolerate simultaneous failure of any combination of $n-k$ nodes while incurring only $\nicefrac{n}{k}$ times of the data block storage on these nodes.
The rule of thumb is that with higher failure rate of the nodes (so they fail more frequently), the storage systems must be able to tolerate simultaneous failures of larger scale, and thus configure with larger placement groups.

Compared to traditional storage systems, one unique challenges of data placement in DSNs stem from the Byzantine nodes and limited governance infrastructure.
The DSNs who are configured with placement groups of $n$ nodes and can tolerate $k$ simultaneous failures of them must ensure that, no more than $f$ Byzantine nodes are selected into the $n$ nodes so that $f+f'>k$, where at most $f'$ benign simultaneous failures are expected from the $n-f$ rational nodes.
Otherwise, those Byzantine nodes may collude with each other and create simultaneous failures that lead to data loss.
The strict security model also forbid DSNs to leverage a trusted single party to manage data placement, but rely on blockchain to reach consensus on the governing routine to tolerate Byzantine nodes.
Blockchains are notorious for high cost of on-chain state and high transaction cost.
In DSNs, every blockchain node is effectively a full ledge storage controller and replicate maintenance effort of all data blocks.
Keeping low on-chain overhead is crucial for DSNs to be scalable and manage large volume of data.

In summary, we want to achieve the following goals of DSNs with their data placement strategies:
\begin{itemize}
    \item Tolerate up to a certain fraction of Byzantine nodes;
    \item Keep low and scalable storage overhead: the storage volume should be proportional to the total storage supply of the nodes;
    \item Minimize the on-chain state footprint.
\end{itemize}

Next, we will explain the data placement strategies employed by these DSNs and how they have caused various drawbacks in the DSNs.

\paragraph{Explicitly store placement on-chain.}
In Filecoin and Sia, the users control placement of their data.
They pick the storing nodes and sign contract with them, and the contracts are stored on-chain as explicit placement metadata.
Only the contracted nodes can make profit by storing the data, and it is mandatory for them to store, or penalty will be applied to them.

The advantage of this per-node explicit record is that it makes erasure coding easy.
Clients can locally encode data blocks and sign contracts with different nodes to store different encoded fragments.
The downside is large on-chain footprint.
The on-chain state for storing each data block is proportional to the number of nodes in the placement group.
To tolerate higher failure rate which is commonly found in DSNs, the on-chain state will be significantly bloated.
Both DSNs have been trying to work around this limitation.
Filecoin chooses large storage unit (32 GiB or 64 GiB) to reduce the number of data, and it incentivizes extremely reliable, cluster-grade nodes to secure stored data with small placement groups of usually less than 10 nodes.
Sia is more suitable for daily personal use with its small storage unit and is able to be more friendly to less reliable nodes with its default placement group size of 30.
This results in overwhelming blockchain state size, and Sia has subsequently shifted to stateless blockchain paradigm (with Utreexo) recently.
As the result, Sia does not permit external clients.
All users must stay online all the time to continuously download new block headers and update the Merkle proofs of their coins, and they have to monitor and audit the nodes storing their data by themselves, as the task is impossible for a stateless blockchain.

What's worse, the number of Byzantine nodes these DSNs can tolerate is proportional to the placement group size instead of the total number of nodes.
The Byzantine nodes can easily attack a data block by offering a low storage price and inducing the user to sign contract with them.
Even if the users choose contracted nodes purely randomly, there is still nontrivial probability that overwhelming Byzantine nodes are chosen initially or throughout the data lifetime when replacing failed nodes.
Thus, the high on-chain overhead of data placement makes these DSNs more vulnerable to Byzantine nodes.

\paragraph{Implicit incentive-driven placement.}
Permacoin and Retricoin are research designs for archiving large volume public static datasets.
They do not enforce data placement but encourage every node to store as much data as possible.
Similar to Bitcoin, nodes are rewarded for successfully mining blocks, which require them to prove that they are storing a certain segment of the datasets.
The more segments a node stores, the high probability for the node to successfully mine blocks and obtain rewards.
Arweave is a production DSN following a similar approach.
The historical blocks in the blockchain are themselves the stored data, and mining blocks of each round requires to prove that a random historical block is stored.
With this approach, Arweave allows the stored data to be dynamically extended.
All these DSNs achieve constant on-chain footprint.
Only a small accumulator (e.g., Merkle root), representing all persisted data, is required to be stored by the nodes and validate mined blocks.
The size of this accumulator remains constant regardless of the stored data size.

Besides lacking function of easily deleting stored data, these DSNs suffer from poor storage efficiency.
Firstly, these DSNs cannot employ erasure coding.

Secondly, similar to the storage contract approach, Byzantine nodes can easily attack a data segment by all storing it and forcing rational nodes to replace it with more profitable data segments.
So, to tolerate Byzantine nodes, the effective placement group size (driven by incentive) must be larger than the number of Byzantine nodes.
Thus, each data segment must be stored on the nodes whose number is proportional to the total number of nodes to tolerate a fraction of Byzantine nodes globally.
This causes high and non-scalable redundancy level which is only feasible when the system scale is small.
In Arweave, 350 TiB data is stored by only around 100 nodes.

\paragraph{Static placement.}
Walrus is an emerging DSN, taking an approach more akin to traditional storage systems.
In Walrus, the top 1000 nodes (according to their staking) are selected as the committee to store all data for each epoch (i.e., two weeks).
A hash function maps each data block to the nodes of its placement group, which collectively store the data with erasure coding.
This design enables both efficient storage and low on-chain footprint, as the on-chain state only needs to store per-data record, and the placement can be solely determined with the hash function.
However, Walrus has limited storage volume because it can only utilize storage resource from 1000 nodes.
The fault tolerance is also limited to a fraction of those utilized nodes.
With sufficient number of nodes in total, a fraction of Byzantine nodes may overwhelm the selected committee and cause data loss.

\paragraph{Randomized replication.}
Swarm designs a data placement strategy that assigns data blocks and nodes into random neighborhoods via a distributed hash table.
The nodes in the same neighborhood stores redundant copies of the same data blocks.
This design achieves constant on-chain footprint.
Data storing and retrieval requests can be routed to the correct neighborhood with the help of the distributed hash table, which does not incur additional on-chain overhead.
To audit the stored data, Swarm relies on game theory and can detect faulty nodes as long as there is at least one honest node in the neighborhood.
However, the game theory-based audit can only be applied with replication-based redundancy, which incurs high storage overhead.
