\section{Motivation}
\label{sec:motivation}

\subsection{DSNs and Their Data Placement}

Decentralized storage networks (DSNs) provide blob storage services backed by resources collectively contributed by permissionless nodes.
Some nodes are rational and aim to generate profit with their storage supply, while others are Byzantine, behaving arbitrarily and potentially maliciously.
DSNs provide reliable storage as long as the share of Byzantine nodes remains below a threshold (e.g., \nicefrac{1}{3}), ensuring stored data can be retrieved.
They incentivize rational nodes to store and serve data correctly while tolerating the presence of Byzantine nodes.

This work focuses on improving data placement strategies in DSNs.
In storage systems, data placement determines, for a given data block, which nodes should store redundant fragments and what form that redundancy takes.
These nodes are the placement group of the data block.
Redundant placement across multiple nodes is crucial for tolerating simultaneous failures.
For instance, storing $r$ copies of a data block on distinct nodes keeps the data available even if $r-1$ nodes fail concurrently.
As long as the storage system can repair the redundancy before all $r$ nodes fail, the storage remains durable.

In practice, storing a large number of full copies incurs excessive storage overhead.
Storage systems commonly employ erasure coding to achieve better storage efficiency.
They encode a data block into $n$ fragments, any $k$ of which can recover the data block, and store them on $n$ nodes.
Then, the storage systems can tolerate simultaneous failure of any combination of $n-k$ nodes while incurring only $\nicefrac{n}{k}$ times the storage of the data block across these nodes.
A common guideline is that higher node failure rates require tolerance of larger simultaneous-failure sets, which in turn motivates larger placement groups.

Compared to traditional storage systems, data placement in DSNs faces additional challenges from Byzantine nodes and limited governance infrastructure.
DSNs configured with placement groups of $n$ nodes that can tolerate $k$ simultaneous failures must ensure that no more than $f$ Byzantine nodes are selected so that $f+f'>k$, where at most $f'$ benign simultaneous failures are expected among the $n-f$ rational nodes.
Otherwise, colluding Byzantine nodes may create simultaneous failures leading to data loss.
Because the security model forbids reliance on a trusted coordinator, DSNs must use blockchains to reach consensus on placement governance to tolerate Byzantine behavior.
Blockchains are notorious for high on-chain state cost and high transaction cost.
In DSNs, every blockchain node effectively acts as a full storage controller and replicates maintenance effort for all data blocks.
Keeping on-chain overhead low is crucial for DSNs to scale and manage large data volumes.

In summary, we want to achieve the following goals of DSNs with their data placement strategies:
\begin{itemize}
    \item Tolerate up to a certain fraction of Byzantine nodes;
    \item Keep low and scalable storage overhead: the storage volume should be proportional to the total storage supply of the nodes;
    \item Minimize the on-chain state footprint.
\end{itemize}

We summarize previous DSNs in \autoref{tab:dsn}, comparing them along three dimensions: storage volume, on-chain footprint, and fault tolerance.
Higher storage volume and fault tolerance are preferable, while lower on-chain footprint is desirable.
Next, we explain the data placement strategies employed by these DSNs and how they introduce specific drawbacks.

\paragraph{Explicit on-chain placement.}
In Filecoin and Sia, users control placement of their data.
They pick storing nodes and sign contracts with them, and the contracts are stored on-chain as explicit placement metadata.
Only the contracted nodes can earn profit by storing the data, and storing the data is mandatory or penalties apply.

Per-node explicit records simplify erasure coding: clients can locally encode data blocks and sign contracts with different nodes to store distinct fragments.
The downside is a large on-chain footprint, because the state for each data block is proportional to the placement group size.
To tolerate the higher failure rates common in DSNs, placement groups must grow, further bloating on-chain state.
Both DSNs attempt to work around this limitation.
Filecoin chooses large storage units (32 GiB or 64 GiB) to reduce the number of tracked data blocks and incentivizes extremely reliable, cluster-grade nodes to secure data with small placement groups, usually under 10 nodes.
Sia targets daily personal use with small storage units and accommodates less reliable nodes with a default placement group size of 30.
This yields overwhelming blockchain state, leading Sia to shift recently to a stateless blockchain paradigm (Utreexo).
As a result, Sia does not permit external clients: all users must remain online to download new block headers, update Merkle proofs of their coins, and monitor and audit the nodes storing their data themselves, since delegation is infeasible in a stateless blockchain.

Moreover, the number of Byzantine nodes these DSNs can tolerate is proportional to the placement group size instead of the total number of nodes.
Byzantine nodes can attack a data block by offering low storage prices and inducing users to sign contracts with them.
Even if users choose contracted nodes purely at random, there remains nontrivial probability that Byzantine nodes dominate initially or during replacement over the data lifetime.
The high on-chain overhead of explicit placement therefore amplifies vulnerability to Byzantine nodes.

\paragraph{Implicit incentive-driven placement.}
Permacoin and Retricoin are research designs for archiving large public static datasets.
They do not enforce data placement but encourage every node to store as much data as possible.
Similar to Bitcoin, nodes are rewarded for successfully mining blocks, which require them to prove storage of specific dataset segments.
The more segments a node stores, the higher its probability of mining a block and earning rewards.
Arweave is a production DSN following a similar approach.
The historical blocks in the blockchain are themselves the stored data, and each mining round requires proving that a random historical block is stored.
This approach allows the stored dataset to be extended dynamically.
All these DSNs achieve constant on-chain footprint.
Only a small accumulator (e.g., Merkle root), representing all persisted data, is required to be stored by the nodes and validate mined blocks.
The size of this accumulator remains constant regardless of the stored data size.

Beyond the difficulty of deleting stored data, these DSNs suffer from poor storage efficiency.
They cannot readily employ erasure coding because miners must prove possession of exact data segments, not coded fragments, to participate in block production.

Furthermore, as in the storage contract approach, Byzantine nodes can attack a data segment by collectively storing it and pushing rational nodes to replace it with more profitable segments.
To tolerate Byzantine nodes, the effective placement group size (driven by incentive) must exceed the number of Byzantine nodes.
Consequently, each data segment must be stored on a number of nodes proportional to the total node population to tolerate a global fraction of Byzantine nodes.
This yields high and non-scalable redundancy, feasible only when the system scale is small.
In Arweave, 350 TiB of data is stored by only around 100 nodes.

\paragraph{Disjoint replication.}
Swarm separates nodes into disjoint neighborhoods based on their overlay-network address prefixes.
Data blocks are uniformly routed, via a distributed hash table (DHT), to randomly selected neighborhoods and then replicated by the nodes within those neighborhoods.
This design achieves a constant on-chain footprint.
Storage and retrieval requests can be routed to the responsible neighborhood through the DHT without additional on-chain overhead.
Swarm does not record stored data on chain; instead, it verifies whether nodes within the same neighborhood store identical data.
Leveraging game-theoretic incentives, Swarm can detect faulty nodes as long as each neighborhood contains at least one honest node.

Swarm suffers from limitations similar to those of DSNs with incentive-driven data placement.
It also cannot readily employ erasure coding.
Although data assignment across neighborhoods is randomized, nodes can freely choose which neighborhood to join.
As a result, Byzantine nodes can cluster in specific neighborhoods and displace rational nodes, eventually rendering those neighborhoods insecure.
Swarm must therefore enlarge neighborhood sizes so that each neighborhood can independently tolerate Byzantine concentration, which incurs substantial storage overhead.

\paragraph{Static placement.}
Walrus is an emerging DSN that more closely resembles traditional storage systems.
In Walrus, the top 1000 nodes (according to their staking) are selected as the committee to store all data for each epoch (of two weeks).
A hash function maps each data block to the nodes of its placement group, which collectively store the data with erasure coding.
This design enables both efficient storage and low on-chain footprint, as the on-chain state only needs to store per-data record, and the placement can be solely determined with the hash function.
However, Walrus has limited storage volume because it can draw resources only from 1000 nodes.
Fault tolerance is likewise limited to a fraction of those utilized nodes.
With a large total node population, a fraction of Byzantine nodes may overwhelm the selected committee and cause data loss.
