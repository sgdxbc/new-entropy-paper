\section{Design}
\label{sec:design}

We propose \sys, a minimal DSN design that instantiates the data placement mechanism introduced in \cref{sec:approach}.
We assume the existence of a token that is economically favorable to rational nodes.
For example, storage costs are more than compensated by storage rewards denominated in this token.

\subsection{On-chain Control Plane}

The smart contract in \sys maintains two on-chain states: the current set of staking nodes and the current set of stored data hashes.
The former determines the number of nodes, which in turn determines the current sample rate $p$.
The smart contract accepts three types of transactions: staking transactions for node joining, storing transactions for clients to store data\footnotemark, and proving transactions for nodes to prove storage and claim rewards.
\footnotetext{For simplicity, we omit data expiration and removal.}

A staking transaction adds a node to the staking set.
A storing transaction adds a data hash to the stored-data set.
At each block height, a random $(\text{node}, \text{data})$ pair is sampled from these states using the block hash as a seed, making the sampled pair unpredictable in advance.
If the sampled node is endorsed in the data placement group, it can submit a proving transaction within a time window, including a VRF proof and the stored fragment.
The smart contract validates the node and data hash in the transaction, then runs $\VRFVerify$ and $\ECVerify$ as described in \cref{sec:approach:ec}.
If validation succeeds, the submitting node receives token rewards.

\subsection{Client Operations}

To store a data block, a client submits a storing transaction with the data hash and the token payment for storage.
When nodes process the storing transaction, they run $\VRFGen$ locally with their $\sk$ and the current sample rate $p$ to determine whether they are endorsed.
If endorsed, they query the client and provide VRF proofs.
The client first runs $\VRFVerify$ to validate these nodes, then encodes fragments using proof hashes as indices and sends the corresponding fragments to them.
The client concludes the store operation after sending a sufficient number of encoded fragments.

To retrieve a data block, a client sends a gossip message to \sys to discover encoded fragments.
Nodes that store relevant fragments return them to the client.
The client validates these fragments in a manner similar to proving transactions and decodes the data block from the validated fragments.
In this work, we omit retrieval incentives.

\subsection{Node Bootstrapping}

Redundancy repair typically occurs naturally when new nodes bootstrap into \sys.
After synchronizing the smart-contract state, a node runs $\VRFGen$ against the current set of stored data hashes.
For each placement group in which the node is endorsed, it queries encoded fragments and decodes the data block, similar to client retrieval.
It then encodes the fragment it is assigned, according to the VRF proof hash, and stores it locally.

\subsection{Configure Sample Rate}

The sample rate $p$ is periodically reconfigured based on Byzantine fault rate $f$, maximum simultaneous benign failures in a placement group $f'$, EC recovery threshold $k$, and the number of nodes $N$.
The parameters $f$, $f'$, and $k$ are preconfigured, and $N$ is derived from the current staking-node state.

If $p$ decreases because $N$ grows, each node reruns $\VRFGen$ on the data corresponding to stored fragments and discards fragments for which it is no longer endorsed.
Conversely, if $p$ increases due to the rare case of system shrinkage, each node reruns the bootstrap routine and stores additional fragments.
According to the random sampling definition (\cref{sec:approach:sample}), nodes only need to discard fragments when $p$ decreases and only need to store more fragments when $p$ increases.
This design minimizes the number of transferred encoded fragments during reconfiguration.
