\section{Implementation}
\label{sec:impl}

In this section, we describe several engineering details beyond \cref{sec:design} that worth mention in our \sys prototype.

\subsection{Accerlate Placement Group Discovery with Distributed Hash Table}

Every put and get operations (including the get operations triggered by repairing) requires the client or bootstrapping node to discover the endorsed nodes to interact with them.
Naively, this can be done by broadcasting to all nodes via a gossip protocol and waiting for responses, but this approach requires $O(N)$ communication overhead and does not scale to large system scales.
To cope with up to hundreds of thousands of nodes, we integrate an efficient node discovery mechanism based on distributed hash table (DHT) into our implementation.

DHT is widely used in peer-to-peer (P2P) systems for locating nodes and data.
In our implementation, we use Kademlia~\cite{kademlia} as the underlying DHT protocol, which organizes nodes and data in a hash space based on their XOR distances.
Kademlia provides a lookup protocol that recursively queries nodes to discover a certain number of closest nodes to a given key.
This protocol yields consistent results on every node, and it can efficiently discover nodes with logarithmic communication overhead in the size of the network.

We adapt the VRF in \sys to cooperate with Kademlia to discover the endorsed nodes for a given data object.
Specifically, we want the endorsed rational nodes to be discoverable by Kademlia.
We define the sample probability of VRF to be a step function and mask off the nodes whose IDs are far from the data block's hash.

To discover the endorsed nodes for a data block, the client or bootstrapping node performs a Kademlia lookup for the block's hash.
The looked up nodes assist the lookup procedure as in the original Kademlia, while the endorsed ones of them piggy back their VRF proof and signature to the lookup response.
The placement group can be fully discovered after the lookup procedure completes in logarithmic rounds.
Then, the client or bootstrapping node can proceed to store to or retrieve from the endorsed nodes as described in \cref{sec:design}.

While the DHT-based approach feels similar to the tranditional P2P storage~\cite{glacier,oceanstore} and prior DSN storage~\cite{swarm}, the key difference is that in \sys the DHT is only used for node discovery, while in prior works the DHT ultimated governs the data placement.
For example, Byzantine nodes can significantly manipulate the data placement in a prior storage system by clustering themselves in the hash space, while in \sys this attack will have no effect as discussed in \cref{sec:approach}.
