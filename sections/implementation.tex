\section{Implementation}
\label{sec:impl}

In this section, we describe several engineering details beyond \cref{sec:design} that are worth highlighting in our \sys prototype.

\subsection{Efficient Placement Group Locating with Distributed Hash Table}

Every put or get operation, including get operations triggered during repair, requires the client or bootstrapping node to discover the endorsed nodes with which it should interact.
A naive approach is to broadcast to all nodes via a gossip protocol and wait for responses, but this method incurs $O(N)$ communication overhead and does not scale to large deployments.
To support up to hundreds of thousands of nodes, we integrate an efficient node discovery mechanism based on a distributed hash table (DHT).

DHTs are widely used in peer-to-peer (P2P) systems to locate nodes and data.
In our implementation, we use Kademlia~\cite{kademlia} as the underlying DHT protocol.
Kademlia organizes nodes and data in a hash space according to XOR distance.
Kademlia's lookup protocol iteratively queries nodes to find a target number of nodes closest to a given key.
The protocol yields consistent results across nodes and discovers candidates with logarithmic communication overhead in network size.
Several prior P2P storage~\cite{oceanstore,glacier} and DSNs~\cite{swarm} directly rely on DHT for data placement due to its desirable communication efficiency and consistency.

However, \sys's data placement is governed by VRF, which enables unpredictable placement and global defense to Byzantine nodes.
The nodes endorsed by VRF uniformly distribute across the hash space, while the nodes discovered by DHT are globally closest to the target.
Thus, DHT cannot feasibly discover all endorsed nodes of a placement group.

To bridge between VRF and DHT, we propose a modified multi-region version of VRF.
For each data block, multiple \emph{subhashes} are deterministically derived from the data hash.
Then, a node is endorsed by the VRF if for any subhash, its node ID is close enough to the subhash considering XOR distance (i.e., it shares a sufficiently long prefix with the subhash), and it is sampled by VRF with the subhash as input, as described in \cref{sec:approach}.
Conceptually, the hash space is divided into multiple regions, each of which contains a subhash, and the VRF samples nodes from each region with it.

This multi-region VRF allows efficient DHT-based node discovery.
We can initiate dedicated lookups per subhash, which conclude after all nodes close enough to the subhash are discovered.
These concurrent lookups have the same $O(\log{N})$ communication overhead as single DHT lookup, significantly reduced from the $O(N)$ gossip baseline.
At the same time, none of the \sys properties is affected by this optimization.
The on-chain state still maintains only the data hash, as the subhashes can be deterministically derived from it on demand.
Most crucially, the security of placement groups is the same as before.
The usage of DHT in \sys is a pure performance optimization of discovering endorsed nodes.
In the end, nodes are still uniformly sampled for endorsement, regardless via a multi-region approach or a single-region one.
Thus, all security analysis we performed in \cref{sec:design:analysis} still applies.
