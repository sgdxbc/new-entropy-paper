In our implementation, client and peer nodes exchange chunks through HTTP endpoints, similar to IPFS's \texttt{add} and \texttt{cat} RPC APIs.
Peers use plain TCP to communicate with each other, with designated public-key signatures on messages when necessary.
The TCP sockets are multiplexed among \sys and DHT's peer discovery service, while fragments are transferred through a bulk transfer service with dedicated connections, isolating from latency-incentive messaging.

The peer selection protocol requires DHT to discover large amount of peers near a chunk hash, e.g. 40-160 in our evaluation.
To cooperate such demand, the concurrency level of contacting peers (\ie, $\alpha$ in Kademlia) has been increased to 30 from its typical setting, \ie, 3.
This configuration can reduce peer discovery latency in a worldwide P2P network, as shown in our evaluation.

We use wirehair \cite{wirehair}, an $O(1)$ time rateless erasure code implementation, as our coding option.
Wirehair not only attains high performance, it can also recover the original data with an expected $k + 0.02$ fragments.

We deploy Tokio~\cite{tokio} as our asynchronous IO runtime and user-space scheduler.
Tokio facilitates highly concurrent IO and efficient parallel encoding.
Our overall implementation is written in 6447 lines of Rust code.
