\section{Implementation}
\label{sec:impl}

In this section, we describe several engineering details beyond \cref{sec:design} that are worth highlighting in our \sys prototype.

\subsection{Efficient Placement Group Discovery with Distributed Hash Table}

Every put or get operation, including get operations triggered during repair, requires the client or bootstrapping node to discover the endorsed nodes with which it should interact.
A naive approach is to broadcast to all nodes via a gossip protocol and wait for responses, but this method incurs $O(N)$ communication overhead and does not scale to large deployments.
To support up to hundreds of thousands of nodes, we integrate an efficient node discovery mechanism based on a distributed hash table (DHT).

DHTs are widely used in peer-to-peer (P2P) systems to locate nodes and data.
In our implementation, we use Kademlia~\cite{kademlia} as the underlying DHT protocol.
Kademlia organizes nodes and data in a hash space according to XOR distance.
Kademlia's lookup protocol iteratively queries nodes to find a target number of nodes closest to a given key.
The protocol yields consistent results across nodes and discovers candidates with logarithmic communication overhead in network size.
Unfortunately, Kademlia clearly cannot feasibly discover all endorsed nodes, which uniformly distributes across the whole hash space.

To address this issue, we modify the endorsement rule to be favorable by DHT.
The endorsed nodes not only need to obtain the sufficiently small pseudorandom output from VRF (\cref{sec:approach:sample}), but also need to have IDs with sufficiently small XOR distances to the data hash, i.e., share a sufficiently long common prefix with the data hash.
Essentially, we only endorse nodes from a restricted \emph{subspace} around the data hash, the nodes of which can be easily discovered with a DHT lookup.
To preserve the expected sample size, we increase the sampling probability according to the common prefix length.
For example, if we endorse nodes whose IDs have 10-bit common prefix with the data hash, the sample rate $p$ is increased by $2^{10}$ times.

At the same time, we also adapt the vanilla Kademlia lookup semantics.
Typically, Kademlia concludes lookups after a fixed number of the closest nodes to the target are discovered.
If we look up endorsed nodes as this, the Byzantine nodes may cluster around the target to be discovered, while the further rational nodes will be overlooked even if they are endorsed.
Instead, in \sys, Kademlia \emph{exhaustively} looks up the target subspaces with uncertain number of nodes.
Specifically, it operates in a neighborhood basis similar to Swarm~\cite{swarm}.
The nodes fully replicate the contacts of nearby nodes and randomly sample nodes from further distance buckets as in vanilla Kademlia.
After a lookup reach any node within the same subspace as the target, the node will respond with the full contact list of the subspace, which is all possibly endorsed nodes.
We adopt standard security treatments \xxx{cite S/Kademlia} such as disjoint lookup paths to defend against eclipse attacks targeting lookup protocol.

Although this DHT-based approach resembles traditional P2P storage~\cite{glacier,oceanstore} and prior DSN systems~\cite{swarm}, a key distinction remains: in \sys, the DHT is used only for node discovery and has nothing to do with data placement.
While the sampling distribution is adjusted from globally uniform to subspace-wise uniform, the node-centric approach (\cref{sec:approach:sample}) remains the same, so as the key property: the rewards to the rational nodes are not affected by the distribution of Byzantine nodes.
Even if all Byzantine nodes reside in a single subspace, the rational nodes of that subspace will still be reliably discovered during the exhaustive lookup, so they will not miss their endorsement and rewards.

In \sys, we increase the parallelism of placement group discovery by using multiple placement groups, which effectively reduces discovery latency.
The client partitions each data block into multiple sub-blocks and can optionally apply erasure coding across them to further improve availability.
Each sub-block is then stored concurrently in \sys with a smaller placement group.
Bounded placement group sizes bound lookup latency, and concurrent lookups are independent and can be parallelized.
As shown in \cref{sec:eval}, employing multiple placement groups yields comparable, and in some cases lower, operation latency.
