\section{Introduction}
\label{sec:intro}

Decentralized storage networks (DSNs) are emerging as a reliable and cost effective approach for long term blockchain data storage.
Blockchain ecosystems generate large volumes of data.
The consensus protocols produce blocks of historical transactions at high rates.
These blocks must be archived for future analysis and auditing.
In addition, blockchain applications store multimedia data for digital assets that have real economic value, so data loss can lead directly to financial loss.
All of this information must be stored reliably under strict trust assumptions in which no single participant is trusted.

DSNs address this requirement.
Unlike centralized storage systems, where one organization owns and manages the resources, DSNs rely on resources contributed by many permissionless providers and coordinated by distributed protocols such as smart contracts executed on blockchains.
Providers can join and leave the network at any time.
DSNs can tolerate data loss on individual nodes caused by node churn, disk failures, and adversarial behavior.
As a result, users need not trust any single provider for availability while retaining confidence that their data remain reliably stored.
DSNs are also cost effective because no single provider can monopolize storage resources.

As in centrally managed distributed storage, the central task in DSNs is to determine how data are placed across nodes.
Each data block is stored redundantly on multiple nodes so that individual node failures do not cause data loss.
For each block, the selected set of storing nodes is the placement group.
DSNs must determine placement groups consistently over the entire lifetime of each block so that data remain retrievable at any time and storage audits can be executed continuously.
Placement groups must also be large enough to tolerate correlated failures and Byzantine nodes, which requires replacing failed group members promptly when failures occur.
Most importantly, DSNs must provide incentives for rational nodes to follow the protocol and store data correctly.
For example, nodes can be required to provide proofs of retrievability~\cite{por}, which prevents them from claiming storage without actually retaining the data.

Compared with traditional distributed storage, DSNs face unique challenges in data placement because of their decentralized infrastructure.
First, a fraction of network nodes is assumed to be Byzantine.
The placement strategy must therefore tolerate Byzantine nodes within placement groups without introducing excessive redundancy.
Second, most DSNs rely on blockchains to govern data placement in order to avoid dependence on any single party.

DSNs must operate under limited on-chain storage capacity and high on-chain operation costs.
Accordingly, they must keep the on-chain metadata overhead of data placement low, even when placement groups are large enough to tolerate high failure rates.
Finally, DSNs must adapt to highly dynamic network conditions.
When nodes leave while storing redundancy for many data blocks, the system must efficiently identify affected blocks and repair lost redundancy by recruiting new nodes into the corresponding placement groups.

Prior DSN designs have focused mainly on enforcing correct behavior by rational nodes, while largely overlooking the efficiency of placement strategies under Byzantine faults.
Filecoin and Sia rely on storage contracts to govern placement, which introduces substantial on-chain storage overhead.
This overhead limits placement group size and therefore reduces tolerance to Byzantine nodes.
They also handle node failures inefficiently, and Sia undesirably shifts the burden of monitoring and repair to clients.
Arweave and Permacoin do not support erasure coding and do not enforce explicit data placement.
As a result, they must store copies on a large fraction of all nodes to tolerate Byzantine behavior, which yields unscalable redundancy.
They also handle node failures inefficiently because individual nodes must scan the network to identify blocks that are profitable to store.
In addition, Walrus underutilizes the storage capacity of non selected nodes and remains vulnerable to Byzantine behavior under its committee based design, while Swarm must replicate data to many nodes to tolerate Byzantine joining attacks.
In summary, existing DSNs do not simultaneously achieve storage efficiency, low on-chain overhead, and efficient repair while tolerating a Byzantine fraction of nodes.
We discuss these DSNs in more detail in \autoref{sec:motivation:review}.

In this work, we propose \sys, an efficient and practical data placement strategy for DSNs.
\sys uses erasure coding to distribute redundancy across large placement groups while maintaining low redundancy.
It stores only a global parameter and one data hash per block on-chain to determine placement groups.
It also assigns new nodes efficiently to the placement groups that require repair.
The core technique in \sys is a verifiable random function (VRF)~\cite{vrf,vrf-rfc} for determining data placement.
A VRF is a cryptographic primitive that maps an input to a pseudorandom output together with a proof of correctness.
The output is unpredictable to anyone without the secret key, yet efficiently verifiable by anyone with the corresponding public key.
Using VRF, \sys determines placement in a manner that adversaries cannot predict or manipulate, without requiring additional consistent on-chain state.
The VRF output also determines the encoded symbols assigned to each node in a placement group.
\sys further employs verifiable rateless erasure coding~\cite{verify-rateless-ec} to generate an unbounded stream of encoded symbols from each original block, so symbols assigned by VRF are distinct with high probability.
These symbols are verifiable using only the data hash and the VRF proof, which enables DSNs to audit storage and incentivize nodes accordingly.
