\section{Introduction}
\label{sec:intro}

Decentralized storage networks (DSNs)~\cite{filecoin,sia,shelby,permacoin,retricoin,arweave,walrus,bft-dsn,swarm} are emerging as a reliable and cost-effective approach to long-term blockchain data storage.
\xxx{Replace the following with real world examples.}
Blockchain ecosystems generate large volumes of data.
Consensus protocols produce blocks of historical transactions at high rates.
These blocks must be archived for future analysis and auditing.
In addition, blockchain applications store multimedia data for digital assets with real economic value, so data loss can directly cause financial loss.

DSNs address the need for reliable storage under strict trust assumptions in which no single participant is trusted~\cite{bar}.
\lijl{Given that the reviewers might not be familiar with the concept, we should elaborate on the motivation of DSN.}\sgd{Done.}
Whereas centralized storage systems focus on tolerating failures of resources operated by a single organization, DSNs assume that organizations themselves may be untrusted and must also tolerate organization-level failures, such as service shutdowns or censorship of stored data.
Consequently, DSNs rely on resources contributed by many permissionless providers and coordinated by distributed protocols such as smart contracts executed on blockchains.
Providers can join and leave the network at any time.
DSNs can tolerate data loss on individual nodes caused by node churn, disk failures, and adversarial behavior.
As a result, users need not trust any single provider for availability while retaining confidence that their data remain reliably stored.
DSNs are also cost-effective because no single provider can monopolize storage resources.

As in centrally managed distributed storage, the central task in DSNs is to determine how data are placed across nodes.
Each data block is stored redundantly on multiple nodes so that individual node failures do not cause data loss.
For each block, the selected set of storing nodes is the placement group.

This placement metadata must be preserved reliably throughout the lifetime of each block to support retrieval and repair.
In some DSNs, smart contracts act similarly to metadata servers in centrally managed storage by maintaining placement state on chain, whereas other DSNs use algorithmic placement and determine placement groups through deterministic algorithms or network protocols.

Data placement in DSNs is particularly challenging because of Byzantine nodes.
Byzantine nodes may behave arbitrarily and are potentially malicious.
Adversaries may attempt to induce data loss and can ignore the economic incentives that motivate rational nodes to follow protocol~\cite{bar}.
In the context of placement, Byzantine nodes are especially dangerous because they can create correlated failures.
Byzantine nodes within the same placement group can collude and discard their stored redundancy simultaneously~\cite{ipfs-attack,bittorrent-attack}.
As a result, placement groups may experience larger simultaneous failures than expected under independent failures, because Byzantine nodes can deliberately amplify the failure scale.
A placement group designed to tolerate $m$ simultaneous failures can fail when just more than $m-f$ non-Byzantine failures occur if the group contains $f$ Byzantine nodes.
Although prior DSN designs have focused mainly on enforcing correct storage by rational nodes and efficiently detecting storage failures~\cite{por}, they largely overlook attacks on data placement, in which no visible failure occurs before data are already lost.

\sgd{Reworked.}
Among placement groups of equal size, groups with fewer rational nodes and more Byzantine nodes are less effective at tolerating failures.
We define the \emph{effective placement group} of a data block as the subset of rational nodes within its DSN-determined placement group.
Data remain available only when this effective placement group is large enough to tolerate simultaneous failures; therefore, in DSNs, the effective placement group is conceptually closer to the placement group in centrally managed storage than the nominal placement group itself.
Unfortunately, DSNs cannot identify Byzantine nodes within placement groups and thus cannot determine effective group size directly.
Moreover, Byzantine nodes can strategically choose which placement groups to join and cluster in selected targets.
Many DSNs incentivize uniform placement-group sizes to balance redundancy.
As the number of Byzantine nodes in a group increases, the number of rational nodes decreases, and the group eventually becomes vulnerable.

\sgd{Reworked.}
For many DSNs~\cite{filecoin,sia,shelby,permacoin,retricoin,arweave}, the only defense against Byzantine clustering in a single placement group is to enlarge groups so that rational nodes outnumber Byzantine nodes.
To tolerate a Byzantine fraction commonly assumed in permissionless systems (e.g., \nicefrac{1}{3})~\cite{bar}, such large groups impose impractical on-chain footprint or constrain total storage volume.
We review limitations of prior DSNs in \cref{sec:motivation:review}.
In summary, prior DSNs do not simultaneously achieve scalable storage volume, low on-chain footprint, and tolerance to a constant Byzantine fraction.

\sgd{Reworked.}
We identify the key factor for efficiently tolerating Byzantine nodes in DSN data placement.
In permissionless decentralized storage, Byzantine nodes can strategically move across placement groups to attack placement outcomes.
Many DSNs defend against this adversarial capability independently within each placement group, similar to how they and centrally managed systems handle benign failures, leading to repeated defensive effort and unfavorable trade-offs.
Instead, a placement mechanism should recognize that the attacker is the \emph{same} Byzantine set across the entire system, with limited total attack capacity that can affect only a limited number of groups at a time.
DSNs can exploit this insight by determining placement groups holistically so that the groups collectively defend against Byzantine concentration.

\sgd{Reworked.}
Following this idea, we design \sys, a DSN in which nodes are randomly sampled into each placement group\footnotemark.
\footnotetext{The name \sys reflects this randomization principle.}
\sys employs a verifiable random function (VRF)~\cite{vrf,vrf-rfc} to enable publicly verifiable sampling, which we call endorsement, while restricting the expected number of groups that endorse each node.
Although Byzantine nodes in \sys can still choose among their sampled groups and, for example, cluster in a single target, the total number of endorsements received by all Byzantine nodes across all groups is VRF-controlled and remains probabilistically bounded.
Exploiting this property, \sys allocates incentives per endorsement rather than per placement group.
When Byzantine nodes cluster in target groups, those groups receive more reward, and the reward received by endorsed rational nodes is unaffected by the Byzantine composition of their groups, so rational nodes remain incentivized to stay.
Consequently, effective placement groups remain stable regardless of how Byzantine nodes distribute themselves, allowing us to analyze data availability using methods similar to those for centrally managed storage in \cref{sec:design:analysis}.
\sys employs erasure coding to ensure scalable storage volume.
We use random sampling again to address encoded-fragment assignment in highly dynamic placement groups with continuous node arrivals and departures.
We adopt verifiable rateless erasure coding~\cite{verify-rateless-ec} to further optimize for space efficiency and enable verifiable storage.

The core technique in \sys is a verifiable random function (VRF)~\cite{vrf,vrf-rfc} for determining data placement.
A VRF is a cryptographic primitive that maps an input to a pseudorandom output together with a proof of correctness.
The output is unpredictable to anyone without the secret key, yet efficiently verifiable by anyone with the corresponding public key.
Using VRF, \sys determines placement in a manner that adversaries cannot predict or manipulate, without requiring additional consistent on-chain state.
\sys further employs verifiable rateless erasure coding~\cite{verify-rateless-ec} to achieve storage efficiency without incurring additional on-chain overhead.
\lijl{Similar to the previous point, should explain the higher level rationale first, before giving the concrete solution.}
The VRF output determines the encoded symbols assigned to each node in a placement group.
Rateless erasure coding generates an unbounded stream of encoded symbols from each original block, so symbols assigned by VRF are distinct with high probability.
These symbols are verifiable using only the data hash and the VRF proof, which enables \sys to audit storage and incentivize nodes accordingly.

We present the complete design of \sys around this novel data placement strategy, including the network protocols for storing and retrieving data, auditing storage, and repairing lost redundancy.
The repairing in \sys only happens when new nodes joining the network and in the rare case of a shrinking network, so it is efficient and does not involve heavyweight on-chain maintenance routines.

In \cref{sec:design}, we present the full design of \sys around this sampling-based placement solution.
We then report preliminary simulation results in \cref{sec:eval}.
\sys preserves sufficient encoded fragments for recovery over a 10-year simulation while incurring only about $2\times$ redundancy.
We also show that \sys tolerates a significantly larger Byzantine fraction than a replication baseline, and that this tolerance improves as redundancy increases.
